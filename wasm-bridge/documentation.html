Ein paraphoner Synth, bei dem jede Qualitätsaussage einen ausführbaren Test hat.

PARA·3 ist ein sound-genauer Software-Synth der KORG-Volca- Keys-Klasse: header-only C++-DSP plus eine verifizierte WASM/ AudioWorklet-Brücke, gebaut, um offline am Handy als PWA zu laufen. Diese Seite dokumentiert ehrlich, wie das Tool entstand, womit und auf welcher Basis geprüft wurde, was herauskam — und wie du es selbst reproduzierst.

01Was PARA·3 ist

Die KORG Volca Keys ist paraphon: drei Sägezahn-Oszillatoren → Mischer → ein gemeinsames Filter → eine gemeinsame Hüllkurve/VCA → digitales Delay. Sechs Voice-Modi verteilen die Oszillatoren neu (Poly, Unison, Octave, Fifth, Uni-Ring, Poly-Ring), LFO moduliert Pitch und Cutoff, ein 16-Step-Sequencer mit Motion- Aufnahme rundet es ab. PARA·3 bildet genau diese Architektur nach.

Technischer Kern: bandbegrenzte Oszillatoren (PolyBLEP, 4× Oversampling, Kaiser-FIR-Dezimierung), ein nichtlineares ZDF-Leiterfilter (TPT, tanh-begrenzte Rückkopplung) mit eigener 2×-Oversampling-Insel, eine 8×-Insel für den Ringmodulator, ein zipper-freier Parameter-Trichter, Denormal-Schutz, und ein lock-freier, doppelt gepufferter Pattern- Speicher. Header-only C++17, framework-agnostisch.

02Wie es entstand

Das Tool wurde sprint-artig in einer durchgehenden Sitzung gebaut. Jeder Schritt produzierte ausführbaren Code und seinen Beweis, bevor der nächste begann:

  • Architektur & sechs Design-Spezifikationen (paraphoner Signalweg, Oszillator, Filter/EG/VCA, LFO/Motion, Delay, Sequencer).
  • DSP-Engine (Para3Engine.hpp) + Mess-Harness (offline_test.cpp, FFT-basiert).
  • C-API-Brücke (RT-sicher, extern "C") + native Verifikation.
  • Lock-freier SPSC-Ring über SharedArrayBuffer + Steuerprotokoll.
  • emcc-Build-Skript, eigenständiger AudioWorkletProcessor.
  • Main-Thread-Glue, COOP/COEP-Server, Steuer-API.
  • Parameter-Trichter auf das volle Panel erweitert (regressionsgeprüft).
  • Anbindung des gestalteten UI, Mobile-Layout, echter Oszilloskop, Logo.
Ehrlich: Während des Baus wurden mehrere Fehler in den Mess-Harnessen gefunden und behoben (z. B. Hann-Fenster- Selbstmaskierung → kohärentes Sampling; O(n²)-Array-Shift im Ringtest; eine nicht-lückenlose Werteannahme bei Backpressure). Die Engine selbst wurde nie „passend gemacht" — gefundene echte Defekte wurden gefixt und neu gemessen. Genau das ist der Punkt der Methode.

03Prüfmethodik & auf welcher Basis

Kein „klingt gut genug". Jede Behauptung ist ein Test mit klarem Kriterium und reproduzierbarer Basis:

DSP-Messung

g++ -O2 -std=c++17 -Wall -Wextra -msse2, kein -ffast-math (IEEE-Präzision). FFT-Analyse mit kohärentem Sampling, damit Anti-Aliasing nicht durch Fenster-Leckage geschönt wird.

Nebenläufigkeit

Node 22 mit echten Worker-Threads + Atomics. Adversariales Interleaving gegen ein Referenzmodell plus ein realer Cross-Thread-Lauf.

Brücken-Treue

Beweis, dass die 128-Sample-Quanten des AudioWorklets bit-identisch zu einem Einzelaufruf sind, und dass der Steuerpfad bit-exakt dekodiert.

Server / Logo

COOP/COEP-Header real per curl bestätigt. Logo bis Favicon-Größe real gerendert und auf Lesbarkeit beider Themes geprüft.

05Was genau an Claude Code übergeben wurde

Nicht loser Text, sondern verbindliche Runbooks mit messbaren Abnahmekriterien und derselben „kein Fake"-Disziplin:

  • README.md — Orientierung & Lese-Reihenfolge.
  • CLAUDE_BUILD.md — Sanity-Gates; Phase A WASM bauen + Gleichwertigkeit messen (kompiliertes WASM == native C-API, Sample-für-Sample); Phase B Offline-Handy: SAB-freier postMessage-Steuerpfad + PWA (Service-Worker) + Mobile-Audio-Unlock; Phase C optional Sprint-1-Hardware-Kalibrierung.
  • CLAUDE_INTEGRATE_UI.md — exakte Tabelle Regler→Engine-Aufruf, plus ehrliche „Liste C": Controls ohne echten Engine-Weg (EG INT, Detune, Portamento, Motion-Ziele ≠ Cutoff) bleiben sichtbar, aber inert — nicht faken.
  • CLAUDE_SCOPE_AND_LAYOUT.md — Mobile-Layout vollständig erreichbar (dvh/svh, einklappbare Tastatur-Schublade, Safe-Area) und ein echter Oszilloskop: read-only AnalyserNode am realen Graph, Stille = flache Linie, plus headless- und In-App-Anti-Fake- Gate (still ⇒ flach und A4 ⇒ 440 Hz).
  • wasm-bridge/ — die geprüfte Brücke + gestaltetes UI; brand/ — das Logo-Set.

06So reproduzierst du es exakt

Voraussetzungen: Node ≥ 18, g++ ≥ 12, Emscripten (emsdk), optional wabt; ein modernes Android-/iOS-Gerät für die Offline-Abnahme.

git clone https://github.com/LEVCON-AT/Para3
cd Para3

# 1 — Sanity (muss grün sein, 0 Warnungen)
g++ -O2 -std=c++17 -Wall -Wextra -msse2 offline_test.cpp -o /tmp/e && /tmp/e
cd wasm-bridge
g++ -O2 -std=c++17 -Wall -Wextra -msse2 -I.. para3_capi.cpp capi_test.cpp -o /tmp/c && /tmp/c
node ring_test.mjs
node audio_test.mjs

Dann das Projekt in Claude Code öffnen und ihm die Runbooks der Reihe nach geben:

  • „Lies CLAUDE_BUILD.md und arbeite es ab" → Build + Gleichwertigkeit + Offline-PWA.
  • „Lies CLAUDE_INTEGRATE_UI.md" → den Entwurf wasm-bridge/para3-responsive.html an die Engine binden.
  • „Lies CLAUDE_SCOPE_AND_LAYOUT.md" → Layout-Fit + echter Oszilloskop.

Abnahme des Ziels: installierte PWA, Flugmodus an, startet und spielt knackfrei; jeder gemappte Regler wirkt hörbar; Scope-Selbsttest PASS; alle Sanity-Tests grün.

07Ehrliche Grenzen

Sprint-1-Kalibrierung. Einige Konstanten (im Code als // CALIB(sprint1) markiert) bestimmen die exakte Klangkopie eines bestimmten physischen Volca-Keys-Geräts. Die Algorithmen sind final und gemessen sauber; die Defaults sind musikalisch gültig. Für „großartiger, spielbarer Offline-Synth" ist das kein Hindernis — für „1:1 nicht unterscheidbar" braucht es eine echte Hardware-Messung (Außenarbeit, kein Code).

Finaler Build. Der eigentliche emcc-Lauf und der Browser-/Gerätetest passieren auf deiner Maschine — ein dokumentierter Befehl. Alle Logik davor ist real gemessen. Der mobile Audio-Stack (besonders iOS-Safari mit AudioWorklet/AnalyserNode) ist erfahrungs- gemäß die fummeligste Stelle; dafür sind die Gerätetests bewusst als harte Abnahme formuliert.
